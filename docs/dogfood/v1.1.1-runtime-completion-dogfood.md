## Deviations

- The first implementation attempt (from a clean baseline) failed: the model reached max iterations (20) without the finalizer triggering, because it had not yet observed a passing test command (it spent iterations resolving a Go toolchain `PATH` issue inside the bash sandbox). The finalizer correctly refused to finalize since `HasPassingTestCommand` was false. The workspace code was reset to baseline (`git checkout`) and the implementation stage was re-run; the second attempt completed in one run via the finalizer. No manual code or artifact edits were made at any point. The reset only removed the model's own incomplete first-attempt output.
- The plan stage required three attempts due to glm-5.2 non-determinism (first: missing `# Technical Plan:` heading; second: stuck in a ToolSearch loop at max iterations; third: passed). The plan stage was not modified by v1.1.1, so this is provider non-determinism, not a regression.
- The model created new files (`recommend.go`, `server.go`, `recommend_test.go`, `server_test.go`) and modified `main.go`, rather than editing the scope-declared files (`tools.go`, `prompt.go`, `main_test.go`). This produced an implementation-review `needs_scope_review` warning (blockers=0), the same advisory level as v1.1. The scope declaration was kept as-is; no manual scope or code edits were made.
- A diagnostics build (temporary stderr logging in `maybeFinalizeRuntimeError`, since reverted) was used to confirm `HasMutation=true` and `HasPassingTestCommand=true` during the finalizer decision. The logging did not alter finalizer behavior; the released build uses the identical decision logic.
