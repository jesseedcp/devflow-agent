## Deviations

- GitHub `workflow_dispatch` requires the workflow file on the default branch, so PR #43 was merged to `main` before the live dispatch. The plan task order was adjusted to merge before the live dogfood; the hard stop (no tag on dogfood failure) was preserved.
- The dogfood demand was initialized to the `deployment` state via a direct `demand.json` edit, mirroring the release-control smoke test. The release-control confirm chain requires artifacts that are not present in a dogfood-only demand.
- `deploy trigger` initially returned `no github actions runs found for workflow release.yml ref main`. Root cause: GitHub registers `workflow_dispatch` runs asynchronously, so the first runs GET (issued before the first poll tick) returned an empty list and `fetchNewestRunAfter` bailed instead of retrying. This was a real defect in the v1.2 poll loop; fixed in commit `8bf46fe` with a sentinel `errNoWorkflowRuns` and retry-until-deadline, plus regression test `TestGitHubActionsDispatchRetriesWhenNoRunsYet`. After the fix, `deploy trigger` returned `deployment pending` and polling reached `passed`.
- `observe refresh` initially failed with `context deadline exceeded`. Root cause: this environment routes `github.com` HTML through a system proxy (`127.0.0.1:21882`) that Go's `http.DefaultClient` does not auto-detect (Go reads only `HTTP(S)_PROXY` env vars, which were unset; `api.github.com` is reachable directly, so `deploy trigger`/`deploy status` were unaffected). Re-running `observe refresh` with `HTTPS_PROXY` set succeeded (200, contains `Release Marker`). No Devflow source change was required; this is an environment proxy configuration, not a code defect.
- After the proxy-caused observation failure moved the demand to `blocked_need_release_decision`, the demand state was reset to `observation` to retry observation (manual state setup, consistent with the plan-permitted manual state setup for the dogfood).
