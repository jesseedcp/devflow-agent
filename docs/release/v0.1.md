# Devflow Agent v0.1 Release Notes

## What v0.1 Contains

- A backend business requirements expert Agent loop from demand input to closeout.
- Local demand workspaces under `.devflow/demands/<demand-id>/`.
- Standard artifacts: `requirements.md`, `plan.md`, `progress.md`, `verification.md`, `closeout.md`, and `memory-candidates.md`.
- Human confirmation gates for requirements, plan, verification, and closeout.
- Quality gate execution through `--quality-command`.
- GitLab MR review gating through `devflow run --stage mr-review`.
- First-run setup through `devflow init`.
- Setup diagnostics through `devflow doctor`.
- Requirements-stage smoke through `devflow smoke`.
- Status and next-action guidance through `devflow status` and `devflow next`.
- Full deterministic dogfood through `devflow dogfood`.
- Version metadata through `devflow version`.
- CI verification and Windows build scripts.
- Optional live sandbox dogfood through `devflow live-dogfood`.
- Direct GitLab MR review checks through `devflow review-gate`.
- Release readiness wrapper through `scripts/release-readiness.ps1`.
- Adds `devflow intake --file <path>` for local PRD intake. It stores `intake.md`, renders `requirements.md`, and stops at the existing requirements review gate.
- Adds `context.md` memory recall for demand intake. `devflow intake` and `devflow recall` now make stable memory and historical candidates visible before requirements confirmation.
- Adds context-aware requirements evaluation for intake/context alignment. `evaluate`, `console`, and `workbench --snapshot` now surface warnings when requirements miss intake bullets or mishandle recalled memory.
- Adds `devflow intake --url <url>` for deterministic HTTP(S) PRD intake. URL intake stores fetched HTML as normalized text, renders `requirements.md`, writes `context.md`, and stops at requirements review.

## Supported First-User Path

```powershell
devflow init --provider openai-compat
devflow start --title "Add coupon eligibility check" --description "Only active members can claim coupons once"
devflow status --demand add-coupon-eligibility-check
devflow next --demand add-coupon-eligibility-check
devflow run --demand add-coupon-eligibility-check --stage requirements
```

## Verification Before Release

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts\release-readiness.ps1 -Version 0.1.0
```

Live provider and GitLab checks are optional private-shell gates. They must not run in default CI unless credentials and target MR state are intentionally provided.

The dogfood script must finish with `state: completed`.

- Wave 12 adds human-gated stable knowledge promotion: `devflow memory list`, `devflow memory promote`, and `devflow memory reject` connect closeout candidates to `.devflow/memory/` reuse.
### Wave 13 - Demand Workspace Operator Loop

- `devflow status --demand <id>` now renders an evidence-aware workspace summary.
- `devflow status --all` lists local demands by attention priority.
- `devflow next --demand <id>` now prefers evidence-aware commands such as confirming PASS verification or reviewing pending memory candidates.


### Wave 14 - Operator Console And Runner Entry

- Adds `devflow console` for an operator-oriented view over local demand workspaces.
- Adds `devflow console --demand <id> --run-next` to execute the next runner-safe agent stage through the existing stage runner.
- Keeps human confirmation, memory decisions, and MR review requirements explicit.

### Wave 15 - Guided Autopilot Drive

- Adds `devflow drive --demand <id>` to execute runner-safe stages until a manual gate.
- Adds `--dry-run` and max-step protection.
- Keeps human confirmation and memory decisions manual.

### Wave 16 - Stage Quality Evaluator

- Adds `devflow evaluate --demand <id>` for deterministic structural stage checks.
- Adds `--stage` filtering and `--strict` script mode.
- Shows concise quality status in `devflow console --demand <id>`.

### Wave 17 - TUI Workbench

- Adds `devflow workbench`, a product demand TUI built on ConsoleSummary.
- Supports demand selection, detail view, refresh, and action shortcuts.

### Wave 18 - Operator Dogfood Hardening

- Adds `devflow dogfood --operator-loop` for offline operator-facing dogfood.
- Adds `devflow workbench --snapshot` for non-interactive workbench evidence.
- Runs operator dogfood during release readiness.

### Wave 19 - Backend Demand Defaults

- Adds `backend_demand` config defaults for quality commands, permission mode, runner/quality roots, and GitLab settings.
- `run`, `console --run-next`, `drive`, and `workbench` shortcuts inherit defaults when flags are omitted.
- `doctor` and `init` surface backend demand defaults; explicit CLI flags still override config.

### Wave 24 - Manual Acceptance Evidence

- Adds `devflow evidence add/list` for local manual acceptance evidence. Operators can record API, log, monitor, manual, or link evidence in `verification.md`; status, console, workbench, and evaluate surface the evidence without auto-confirming verification.

### Wave 25 - GitHub CI Gate

- Adds `devflow ci-gate --github-repo <owner/repo> --github-pr <number>` for direct GitHub PR CI status checks.
- `devflow run --stage mr-review` can optionally include GitHub CI status through `--github-repo` and `--github-pr`; pending or failing CI blocks the move to verification.

### Wave 26 - GitHub PR Review Gate

- Adds GitHub PR review-thread support to `review-gate` and `run --stage mr-review` via `--review-provider github` with `--github-repo` and `--github-pr`.
- GitHub unresolved review threads now map into the same review action plan as GitLab comments, and can compose with the GitHub CI gate through `--github-ci`.

### Wave 27 - Provider-Neutral Change-Request Terminology

- Adds provider-neutral `change-request` (alias `cr`) CLI command and `ChangeRequest*` type/option aliases while preserving `mr` and `mr-review` compatibility.
- The engine prefers the new `ChangeRequest` option when set and falls back to the legacy `MergeRequest` option otherwise.
## Known Limits

- GitLab comment replies, resolving discussions, reviewer assignment, and approvals are still external.
- Live provider smoke tests are opt-in and require local API credentials.
- The first product target is backend demand flow only; frontend, test, and PD Agents remain future extensions.
- Intake is local-file only in Wave 20. URL, WeChat, Aone, and DingTalk document intake remain future adapter work.
- Memory recall is deterministic keyword search in Wave 21. Embeddings, ranked semantic search, and external wiki adapters remain future work.
- Context-aware evaluation is deterministic text matching in Wave 22. It is a review signal, not semantic proof that requirements are complete.
- URL intake in Wave 23 only fetches directly accessible static HTTP(S) pages. It does not handle authenticated enterprise portals, JavaScript-only rendering, WeChat anti-scraping, or document-platform APIs.
- Manual evidence in Wave 24 stores operator-provided references only. It does not query log, monitor, API, or screenshot systems automatically.
- GitHub CI gating in Wave 25 reads check runs only. It does not query legacy commit statuses, reply to PR review comments, or create GitHub pull requests. Default release readiness stays offline and does not require `GITHUB_TOKEN`.
- GitHub PR review gating in Wave 26 reads unresolved review threads only. It does not reply to or resolve GitHub review threads, and it does not create GitHub pull requests. Default release readiness stays offline and does not require `GITHUB_TOKEN`.
- Wave 27 terminology is additive only. The `change-request` command wraps the existing GitLab merge-request implementation, and the `mr_review` workflow state name is intentionally unchanged.
