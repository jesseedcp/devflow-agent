# Devflow v0.9.0

## Release Status

v0.9.0 introduces the release-control loop. It is verified offline through `scripts\release-readiness.ps1 -Version 0.9.0`, which covers deploy, observe, and rollback-decision surfaces through a local fake GitHub Actions API. No live GitHub token is required for the default readiness gate.

## What v0.9 Contains

- A `deployment` workflow stage entered after verification is confirmed.
- An `observation` stage that confirms post-deploy evidence before closeout.
- Human-gated rollback decisions recorded in `rollback.md`.
- GitHub Actions as the deployment provider.
- Release state surfaced in `status`, `console`, `workbench --snapshot`, and next-action output.
- Release evidence enforced by `evaluate --stage deployment` and `evaluate --stage observation`.
- Offline release-readiness smoke for deploy, observe, and rollback.

## Release Control Loop

After verification is confirmed, Devflow enters `deployment` instead of `closeout`. A demand cannot reach `closeout` until deployment and observation pass, or a rollback decision is recorded.

```powershell
$env:GITHUB_TOKEN = "<github token>"
devflow deploy trigger --demand add-coupon-eligibility-check --provider github --github-repo "owner/repo" --workflow "release.yml" --ref "main"
devflow deploy status --demand add-coupon-eligibility-check --provider github --github-repo "owner/repo" --workflow "release.yml" --ref "main"
devflow observe refresh --demand add-coupon-eligibility-check
```

If deployment or observation fails, Devflow records a rollback recommendation and blocks in `blocked_need_release_decision` until an operator records a decision:

```powershell
devflow rollback plan --demand add-coupon-eligibility-check --trigger "deployment failed" --impact "release blocked" --recommendation "redeploy after fix"
devflow rollback confirm --demand add-coupon-eligibility-check --decision risk_accepted --by "<name>" --summary "<summary>"
```

`risk_accepted` advances to closeout; `redeploy_required` returns the demand to `deployment`.

## GitHub Actions Support

GitHub Actions is the only deployment provider in v0.9. `deploy trigger` dispatches a workflow run and records the newest run as deployment evidence. `deploy status` refreshes deployment evidence without dispatching. Observation reads the deployment evidence and records a passing observation only when the deployment run completed successfully.

## Rollback Decision Model

v0.9.0 records rollback decisions only. It does not execute rollback. `rollback plan` writes a recommendation; `rollback confirm` records a human decision (`rollback_confirmed`, `risk_accepted`, or `redeploy_required`). The decision is stored in `rollback.md` and surfaced in operator views.

## Verification Before Release

`evaluate --stage deployment --strict` and `evaluate --stage observation --strict` enforce release evidence. Closeout evaluation blocks when observation has not passed and no rollback decision has been accepted. A `rollback_confirmed` decision does not satisfy closeout on its own.

## Known Limits

- GitHub Actions is the only deployment provider.
- Observation uses release-provider evidence only.
- No SLS/Grafana/Datadog/Prometheus/OpenTelemetry adapter.
- Rollback execution is not automated.
- Default release-readiness uses fake providers.
