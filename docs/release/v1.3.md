# Devflow v1.3

Devflow v1.3 adds the platform server and MySQL foundation: a Docker MySQL
store, a `devflow server` command, a migration runner, the core platform
schema, and the first read APIs backed by the on-disk `.devflow` artifacts.

## Shipped

- `docker-compose.yml` runs MySQL 8.4 with database `devflow`, user `devflow`,
  password `devflow`. The host port is published as `3316:3306`.
- `.env.example` documents `DEVFLOW_DATABASE_DSN` and the dev-mode identity
  variables consumed by the server.
- `devflow server` command with flags:
  - `--addr` (default `127.0.0.1:8080`)
  - `--root` (project root)
  - `--database-dsn` (falls back to `DEVFLOW_DATABASE_DSN`)
- Startup validates the DSN, opens MySQL, runs migrations, and starts the HTTP
  server. The DSN/password is never logged.
- `internal/platform/store/mysql` migration runner (`schema.sql` +
  `migrations.go`) creates all platform tables with `CREATE TABLE IF NOT
  EXISTS`.
- `internal/platform/store` model types: `User`, `Workspace`,
  `WorkspaceMember`, `DemandIndex`, `AuditEvent`, plus `Role` and `Action`.
- `server.Store` is a consumer-defined interface in
  `internal/platform/server/server.go`; `*mysql.Store` satisfies it
  structurally. Server tests use an in-memory `fakeStore`.
- `GET /api/health` returns `{"status":"ok","database":"ok"}`.
- `GET /api/me` returns the dev-mode current user.
- `GET /api/workspaces` lists workspaces; `POST /api/workspaces` creates one.
- Artifact bridge (`internal/platform/artifactbridge`) scans `.devflow/demands`
  on disk and exposes:
  - `GET /api/workspaces/{workspaceID}/demands`
  - `GET /api/workspaces/{workspaceID}/demands/{demandKey}`
  - `GET /api/workspaces/{workspaceID}/demands/{demandKey}/artifacts/{name}`
- Artifact reads are gated by an allowlist (`artifactbridge.allowedArtifacts`)
  and reject path traversal.
- API contract types in `internal/platform/api/types.go`:
  `HealthResponse`, `CurrentUser`, `Workspace`, `ArtifactSummary`,
  `DemandSummary`, `EvidenceSummary`, `ReleaseSummary`, `DemandDetail`,
  `AuditEvent`, `ErrorResponse`.

## Schema

The migration creates five tables:

- `users` — platform identity (`id`, `email`, `display_name`, `created_at`).
- `workspaces` — team workspace owning an artifact root
  (`id`, `name`, `artifact_root`, `created_at`).
- `workspace_members` — links a user to a workspace with a role
  (`workspace_id`, `user_id`, `role`, `created_at`).
- `demands` — index rows for UI listing/search; the durable record stays on
  disk under `.devflow`
  (`id`, `workspace_id`, `demand_key`, `title`, `state`, `attention`,
  `artifact_path`, `updated_at`).
- `audit_events` — compliance audit records
  (`id`, `workspace_id`, `actor_user_id`, `action`, `subject_type`,
  `subject_id`, `metadata_json`, `created_at`).

## Port Deviation

The plan specified `3306:3306`, but the host runs a native MySQL 8.4 on port
`3306`. Docker could not bind the host port, so the compose service publishes
`3316:3306` instead. The default DSN in `.env.example` and the test fallback
DSN both use `127.0.0.1:3316`.

## Not Shipped

- No full authentication; identity is dev-mode only via env vars.
- No wiki, rollback execution, or observation adapters (later releases).
- No React frontend (v1.5).
