# Devflow v1.4

Devflow v1.4 adds RBAC, dev-mode identity, and audit on top of the v1.3
platform foundation. Protected actions are gated by role and every protected
action is recorded as an audit event.

## Shipped

- Dev-mode identity is read from env at startup and stored in
  `server.Config{DevUserEmail, DevUserRole, DevUserID:"devlocal"}`.
  - `DEVFLOW_DEV_USER_EMAIL` (default `dev@devflow.local`)
  - `DEVFLOW_DEV_USER_ROLE` (default `Admin`; unknown values fall back to
    `Viewer` so a misconfigured identity can never gain elevation)
- Four cumulative roles in `internal/platform/store/rbac.go`:
  `Viewer` < `Developer` < `Reviewer` < `Admin`.
- `store.Can(role, action)` is a pure, dependency-free permission check.
  Unknown roles or actions return `false` (deny by default).
- Role permission ladder:
  - **Viewer** — read demands.
  - **Developer** — Viewer + add evidence, refresh plan context, refresh
    implementation review.
  - **Reviewer** — Developer + confirm gate, promote wiki, reject wiki.
  - **Admin** — Reviewer + configure workspace, trigger deploy, trigger
    rollback, accept release risk.
- `server.requireAction` middleware enforces RBAC before a protected handler
  runs; forbidden requests get HTTP 403 with a descriptive message. The role is
  read from `Config`, not per-request env.
- `store.NewAuditEvent` builds an audit record with metadata marshaled to JSON.
  `server.recordAudit` appends the event for the current dev user. Audit write
  failures are logged but do not undo an already-completed action.
- `POST /api/workspaces` is the first protected, audited endpoint: gated by
  `configure_workspace` (Admin only) and records a `configure_workspace`
  audit event on success.
- `GET /api/workspaces/{workspaceID}/audit` returns audit events for a
  workspace with metadata parsed from JSON into an object for the frontend.

## Audit Actions

`internal/platform/store/audit.go` labels the high-risk actions:
`confirm_gate`, `trigger_deploy`, `trigger_rollback`, `accept_release_risk`,
`promote_wiki`, `reject_wiki`, `configure_workspace`.

## Not Shipped

- Full authentication and real session identity (still dev-mode).
- Wiki, deploy, rollback, and observation handlers (later releases). Only
  `configure_workspace` is wired as a live protected action so far.
- Frontend role-aware views (v1.5).
