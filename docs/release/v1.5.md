# v1.5.0 React/Vite Web UI Shell

Release scope: frontend lane of the platformization master plan (Tasks 5 and 6),
plus the frontend half of Task 7 (wiki HTTP client wiring and wiki/candidates pages).

## What shipped

- Vite + React + TypeScript app scaffolded under `web/`.
- API type layer (`web/src/api/types.ts`) mirroring the backend contract in
  `docs/superpowers/specs/2026-07-07-platformization-server-webui-design.md`.
- `ApiClient` interface with `mockClient` and `httpClient` implementations,
  switchable via `VITE_DEVFLOW_API_MODE` (`mock` default, `http` for real API).
- Operational pages (no marketing/landing surface):
  - `/workspaces` — workspace list
  - `/workspaces/:workspaceId/demands` — demand list with state/attention badges
  - `/workspaces/:workspaceId/demands/:demandKey` — demand detail with release
    line, quality gate, acceptance evidence counts, metrics summary, and
    artifact tabs (on-demand content fetch)
  - `/workspaces/:workspaceId/wiki` — promoted wiki library
  - `/workspaces/:workspaceId/wiki/candidates` — candidate review with
    role-gated promote/reject (Reviewer/Admin)
  - `/workspaces/:workspaceId/release/:demandKey` — release decision panel with
    Admin-gated rollback trigger and Developer+ observation refresh
  - `/workspaces/:workspaceId/audit` — audit event timeline
- `AppShell` with left nav, workspace/role strip, and a mock-mode role switcher
  so RBAC action states can be demonstrated without a backend.
- Role-aware disabled actions via `RoleGate` and `hasRole`; every high-risk
  action surfaces a clear blocking reason.
- Dense, responsive operational stylesheet (desktop + mobile breakpoints).

## Mock fixtures

Match the required contract: one workspace, three demands in different states
(observation / mr_review / blocked_need_user), one promoted wiki entry, two
pending wiki candidates, one rollback-needed release summary, and five audit
events. Mock client is stateful: promote/reject, rollback trigger, and
observation refresh mutate in-memory state and append audit events.

## Integration assumptions (waiting on backend)

The frontend is built mock-first. When the backend lane lands, set
`VITE_DEVFLOW_API_MODE=http` (and optionally `VITE_DEVFLOW_API_BASE`) to use
`httpClient`. Assumptions for the integration lane to confirm:

- `GET /api/me` returns the dev-mode `CurrentUser` (id, email, displayName,
  role). If the backend folds this into `/api/health` instead, update
  `httpClient.getCurrentUser`.
- Wiki routes match Task 7: `GET /api/workspaces/{id}/wiki`,
  `GET .../wiki/candidates`, `POST .../candidates/{cid}/promote`,
  `POST .../candidates/{cid}/reject`.
- Release routes: `GET .../release/{demandKey}`,
  `POST .../release/{demandKey}/rollback/trigger`,
  `POST .../release/{demandKey}/observe`.
- Artifact content endpoint returns raw text (`text/plain`).

## Known limits

- No real authentication; role comes from dev-mode identity. The mock role
  switcher is mock-mode only.
- Static file serving from the Go server is deferred to the integration lane.
- No screenshots captured in this lane; render correctness is covered by
  component/page smoke tests instead.
