# v0.2 Direction Note

## Context

v0.1 delivers the backend-demand loop with local/URL intake, deterministic verification, operator surfaces, manual evidence, GitLab change-request support, and GitHub change-request support.

## Option A: Platform Integrations

Build adapters for enterprise intake and evidence systems:

- Feishu / DingTalk / WeChat / Confluence / Jira / Aone intake
- authenticated PRD pages
- log and monitoring evidence adapters
- API test evidence import

Benefits:

- Makes Devflow fit real enterprise demand sources.
- Reduces manual copy-paste into intake and evidence.

Risks:

- Higher credential and environment complexity.
- More provider-specific edge cases.

## Option B: Review Automation

Deepen GitHub/GitLab review operations:

- reply to review comments
- resolve threads
- request reviewers
- manage labels and draft state
- read legacy commit statuses
- optional auto-merge gates

Benefits:

- Makes the change-request loop more autonomous.
- Builds directly on Waves 25-28.

Risks:

- More side-effectful operations.
- Requires stronger safety controls before automation.

## Recommendation

Choose Option A if the next pain is getting real business requirements and acceptance evidence into Devflow.

Choose Option B if the next pain is reducing operator work during PR/MR review.

Recommended v0.2 default: Platform Integrations first, with a small review-automation spike for safe read-only legacy status support.

## Dogfood Signals

- If GitHub dogfood required too much manual copying from PRD/evidence systems, platform integrations should win.
- If GitHub dogfood was smooth until review comments or PR metadata operations, review automation should win.
- If both were painful, choose one platform integration and one read-only review automation improvement, not two write-heavy features.

The v0.1 GitHub dogfood was smooth on the read-only and PR-creation surface: PR ensure/create, reuse, review gate, and CI gate all worked directly through the CLI with only a private-shell token. The main friction observed was upstream of GitHub, in getting real requirements and acceptance evidence into the loop (the local runner still needs substantial manual/agent effort). This signal leans toward Platform Integrations first, consistent with the recommended v0.2 default.
