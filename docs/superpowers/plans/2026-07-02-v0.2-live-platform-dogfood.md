# v0.2 Live Platform Dogfood Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Prove the merged v0.2 platform integrations work against real GitHub Issues and Feishu test assets, then record a sanitized dogfood report without committing secrets.

**Architecture:** This is a release-hardening dogfood lane, not a feature lane. Run the existing Devflow CLI on `main`, use isolated local dogfood roots under `.devflow-live-dogfood/`, write only sanitized evidence to docs, and keep all live tokens in shell environment variables.

**Tech Stack:** Go CLI (`go run ./cmd/devflow` or built binary), PowerShell, GitHub Issues API through Devflow, Feishu tenant-token/Doc/Bitable APIs through Devflow, existing `docs/release/v0.2.md`, no new dependencies.

---

## Scope

In scope:

- Verify local `main` is clean except known unrelated untracked files.
- Verify platform doctor behavior with and without credentials.
- Run live GitHub Issue intake.
- Run live GitHub Issue sync and verify marker-based comment behavior.
- Run live Feishu Doc intake.
- Run live Feishu Bitable pool list.
- Run live Feishu Bitable record intake.
- Run live Feishu Bitable sync and verify record fields are updated.
- Run default offline release readiness again.
- Write a sanitized dogfood report.
- Optionally commit the sanitized report only.

Out of scope:

- No new product code.
- No test fixture rewrite.
- No real secret values in git, logs, docs, reports, screenshots, or terminal transcripts.
- No automatic closing of GitHub Issues.
- No destructive Feishu writes beyond updating the explicit test Bitable record.
- No pushing directly to `main` unless the dogfood report is intentionally committed and pushed through a PR.
