# v1.6.1 Platform UI Browser Dogfood Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Prove the v1.6.0 platform Web UI works in a real browser against a live Devflow API server and Docker MySQL, then fix only integration or visual defects found by that dogfood.

**Architecture:** v1.6.1 is a hardening and dogfood release, not a feature expansion. It starts local Docker MySQL on host port `3316`, starts `devflow server` against a temporary `.devflow` demand root, runs the React/Vite Web UI in HTTP mode, verifies browser-visible pages at desktop and mobile sizes, records evidence in `docs/dogfood/v1.6.1-platform-ui-browser-dogfood.md`, and optionally fixes small API/CORS/layout defects discovered during the dogfood.

**Tech Stack:** Go, Docker MySQL, Devflow platform server, React/Vite/TypeScript, browser automation through Playwright or Chrome/Browser Plugin when available, PowerShell release-readiness, GitHub CLI for PR/release.

## Global Constraints

- Do not add product scope beyond platform UI browser dogfood and hardening.
- Do not add rollback execution.
- Do not add observability adapters.
- Do not add external wiki providers.
- Do not change the MySQL port decision; Docker MySQL remains on host port `3316`.
- Do not delete or migrate `.devflow` artifacts into MySQL.
- Do not edit old unrelated untracked plan files.
- Keep Web UI mock mode working.
- Keep Web UI HTTP mode working against `http://127.0.0.1:8080`.
- Any code fix must be caused by a reproduced dogfood defect and verified before commit.
