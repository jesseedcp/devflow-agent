# v0.2 Platform Integrations Design

## 1. 背景

v0.1 已经完成后端需求交付闭环：

```text
intake / start
-> requirements
-> plan
-> implementation
-> mr-review / change-request review
-> verification
-> closeout
-> memory candidates
```

同时，v0.1 已经支持 GitLab 和 GitHub 的 change-request 路径：

- GitLab MR ensure/create。
- GitLab unresolved discussion review gate。
- GitHub PR ensure/create。
- GitHub unresolved review-thread gate。
- GitHub PR CI gate。
- Provider-neutral `change-request` / `cr` 命令别名。

这些能力让 Devflow 能跑完本地和代码评审侧闭环，但真实业务需求进入 Devflow 仍然偏手工：

- GitHub Issue 还不能直接作为 demand intake 来源。
- 飞书文档还不能直接作为 PRD intake 来源。
- 飞书多维表格还不能作为需求池和状态看板。
- 平台鉴权、权限检查和 doctor 还没有统一模型。

这与《如何搭建一个端到端业务需求专家 Agent》中的产品方向一致：代码生成不是瓶颈，真正的瓶颈是上下文组织、工具串联和反馈沉淀。v0.2 的目标不是做更多代码自动化，而是补齐“上下文输入层”和“协作留痕层”的平台入口。

## 2. 产品目标

v0.2 的目标是让 Devflow 能从 GitHub 和飞书承接真实需求，并把关键进展同步回原平台。

### 2.1 GitHub 目标

用户可以从 GitHub Issue 创建 demand：

```powershell
devflow intake --github-issue jesseedcp/devflow-agent#123
```

Devflow 会读取：

- issue title
- issue body
- labels
- assignees
- issue comments
- issue URL 和 metadata

然后生成：

```text
.devflow/demands/<id>/intake.md
.devflow/demands/<id>/context.md
.devflow/demands/<id>/requirements.md
```

用户也可以把 Devflow 的进度同步回 GitHub Issue：

```powershell
devflow sync --github-issue jesseedcp/devflow-agent#123 --demand <id>
```

同步内容包括：

- requirements 摘要。
- plan 摘要。
- change request / PR 链接。
- verification 摘要。
- closeout 摘要。

### 2.2 飞书文档目标

用户可以从飞书 PRD 文档创建 demand：

```powershell
devflow intake --feishu-doc <doc-url-or-token>
```

Devflow 会读取飞书文档正文，转成稳定 Markdown 文本，保存到 `intake.md`，并进入现有 requirements review gate。

v0.2 的飞书文档能力只读，不直接改写原文档。

### 2.3 飞书多维表格目标

用户可以把飞书多维表格当成需求池：

```powershell
devflow pool list --feishu-bitable <app-token> --table <table-id>
devflow intake --feishu-bitable <app-token> --table <table-id> --record <record-id>
devflow sync --feishu-bitable <app-token> --table <table-id> --record <record-id> --demand <id>
```

一条飞书记录对应一个外部需求。Devflow 可以：

- 列出需求池记录。
- 从某条记录创建 demand。
- 把 demand 状态、verification、closeout 同步回该记录。

## 3. 非目标

v0.2 不做以下内容：

- 不接钉钉、企业微信、Jira、Aone、Confluence。
- 不接日志平台、监控平台、API 测试平台。
- 不自动写长期 wiki。
- 不自动更新 codemap。
- 不自动回复或 resolve GitHub review comments。
- 不自动 merge PR。
- 不做 Agent Team。
- 不把 `mr_review` 工作流状态重命名。
- 不在默认 release-readiness 中调用真实 GitHub 或飞书。

这些能力可以作为 v0.3 之后的方向。

## 4. 设计原则

### 4.1 平台 API 不进入 demandflow

`internal/demandflow` 只处理需求、上下文、阶段、产物、事件和下一步动作。它不应该知道 GitHub Issue JSON 或飞书文档块结构。

平台能力放到新边界：

```text
internal/platform
internal/platform/github
internal/platform/feishu
```

### 4.2 统一模型优先

GitHub 和飞书 adapter 都要输出统一模型：

```text
IntakeSnapshot
ExternalDemand
ProgressUpdate
CloseoutUpdate
PlatformCapability
```

主流程只消费这些统一模型。

### 4.3 先读，后写，写操作要可追踪

v0.2 中写操作只做两类：

- GitHub Issue comment 同步。
- 飞书多维表格状态字段更新。

所有写操作必须：

- 支持 dry-run。
- 写入 Devflow event。
- 使用去重 marker 或外部 ID，避免重复刷屏。
- 不在 release-readiness 默认路径中执行真实网络写操作。

### 4.4 鉴权集中管理

平台 token 不写入仓库，不写入 `.devflow/config.yaml`。

配置文件只保存非密信息，例如默认 repo、飞书 app id、表格字段映射。secret 从环境变量读取。

### 5.2 统一类型

`internal/platform/types.go`：

```go
package platform

import "time"

type SourceKind string

const (
	SourceGitHubIssue  SourceKind = "github_issue"
	SourceFeishuDoc    SourceKind = "feishu_doc"
	SourceFeishuRecord SourceKind = "feishu_bitable_record"
)

type IntakeRef struct {
	Kind       SourceKind
	Repo       string
	Issue      string
	URL        string
	Token      string
	AppToken   string
	TableID    string
	RecordID   string
	BaseURL    string
	Properties map[string]string
}

type IntakeSnapshot struct {
	Kind       SourceKind
	ExternalID string
	Title      string
	Body       string
	URL        string
	Author     string
	Labels     []string
	Comments   []ExternalComment
	Metadata   map[string]string
	FetchedAt  time.Time
}

type ExternalComment struct {
	ID        string
	Author    string
	Body      string
	URL       string
	CreatedAt time.Time
}

type ExternalDemand struct {
	ID          string
	Title       string
	Description string
	Status      string
	Priority    string
	Owner       string
	URL         string
	Metadata    map[string]string
}

type ProgressUpdate struct {
	DemandID string
	Stage    string
	Summary  string
	URL      string
	Marker   string
}

type CloseoutUpdate struct {
	DemandID     string
	Result       string
	Verification string
	Knowledge    string
	URL          string
	Marker       string
}
```

### 5.3 统一接口

```go
type IntakeAdapter interface {
	FetchIntake(ctx context.Context, ref IntakeRef) (IntakeSnapshot, error)
}

type DemandPoolAdapter interface {
	ListDemands(ctx context.Context, ref IntakeRef) ([]ExternalDemand, error)
	FetchDemand(ctx context.Context, ref IntakeRef) (ExternalDemand, error)
	UpdateDemand(ctx context.Context, ref IntakeRef, update DemandStatusUpdate) error
}

type PlatformSyncAdapter interface {
	PostProgress(ctx context.Context, ref IntakeRef, update ProgressUpdate) error
	PostCloseout(ctx context.Context, ref IntakeRef, update CloseoutUpdate) error
}
```

`DemandStatusUpdate`：

```go
type DemandStatusUpdate struct {
	DemandID     string
	State        string
	Stage        string
	Summary      string
	Verification string
	Closeout     string
	URL          string
}
```

## 6. GitHub Issue Intake

### 6.1 命令

```powershell
devflow intake --github-issue owner/repo#123
```

也支持拆开参数：

```powershell
devflow intake --github-repo owner/repo --github-issue 123
```

### 6.2 行为

Devflow 调 GitHub REST API：

- `GET /repos/{owner}/{repo}/issues/{issue_number}`
- `GET /repos/{owner}/{repo}/issues/{issue_number}/comments`

只读取 issue，不读取 PR，因为 GitHub PR 也是 issue，adapter 必须拒绝 pull request 类型的 issue。

如果 GitHub 返回的 issue 包含 `pull_request` 字段，返回错误：

```text
github issue intake requires a GitHub Issue, got pull request #<n>
```

### 6.3 生成内容

`intake.md`：

```markdown
# Intake: <issue title>

Source: github_issue
URL: <issue html url>
External ID: owner/repo#123

## Issue Body

<issue body>

## Labels

- label-1
- label-2

## Comments

### <author> at <time>

<comment body>
```

`context.md` 增加来源摘要：

```markdown
## External Source

- Kind: github_issue
- ID: owner/repo#123
- URL: <issue url>
```

### 6.4 Demand ID

默认 demand id 从 issue title slug 生成。若冲突，沿用当前 slug/hash 机制。

## 7. GitHub Issue Sync

### 7.1 命令

```powershell
devflow sync --github-issue owner/repo#123 --demand <id>
```

### 7.2 同步内容

按当前 demand 状态生成一条平台进展 comment：

```markdown
<!-- devflow-sync:demand=<id>:stage=<stage> -->

## Devflow Update: <stage>

State: <state>

Summary:
<summary>

Artifacts:
- requirements.md
- plan.md
- verification.md
- closeout.md
```

如果同一个 marker 已存在，优先 PATCH 更新原 comment；如果 GitHub API 不允许更新或找不到 marker，则创建新 comment。

### 7.3 Closeout Sync

需求完成后，`sync` 发送 closeout 摘要：

```markdown
<!-- devflow-sync:demand=<id>:stage=closeout -->

## Devflow Closeout

Result:
<closeout result>

Verification:
<verification summary>

Knowledge candidates:
<memory candidate summary>
```

v0.2 不自动 close issue。关闭 issue 是单独显式参数：

```powershell
devflow sync --github-issue owner/repo#123 --demand <id> --close-external
```

`--close-external` 必须只在 demand state 为 `completed` 时生效。

## 8. 飞书文档 Intake

### 8.1 命令

```powershell
devflow intake --feishu-doc <doc-url-or-token>
```

### 8.2 鉴权

读取环境变量：

```text
FEISHU_APP_ID
FEISHU_APP_SECRET
```

使用飞书 tenant access token。token 缓存在进程内，不写入磁盘。

### 8.3 文档解析

飞书文档 adapter 负责：

- 从 URL 解析文档 token。
- 拉取文档标题。
- 拉取文档块内容。
- 将段落、标题、列表、表格转为 Markdown。

v0.2 必须支持：

- 标题。
- 普通段落。
- 有序列表。
- 无序列表。
- 简单表格。
- 代码块按纯文本保留。

v0.2 可以降级处理：

- 图片以 `[image: alt or token]` 占位。
- 附件以 `[attachment: name]` 占位。
- 复杂富文本样式忽略。

### 8.4 输出

输出同 GitHub intake：

```text
intake.md
context.md
requirements.md
```

`intake.md` 头部：

```markdown
# Intake: <doc title>

Source: feishu_doc
URL: <doc url>
External ID: <doc token>
```

## 9. 飞书多维表格需求池

### 9.1 配置

`.devflow/config.yaml` 只保存非密配置：

```yaml
platforms:
  feishu:
    app_id: cli_xxx
    bitable:
      default_app_token: appxxxx
      default_table_id: tblxxxx
      fields:
        title: 需求标题
        description: 需求描述
        status: 状态
        priority: 优先级
        owner: 负责人
        devflow_demand_id: Devflow Demand ID
        devflow_state: Devflow 状态
        verification: 验收摘要
        closeout: 交付总结
```

`FEISHU_APP_SECRET` 从环境变量读取。

### 9.2 命令

```powershell
devflow pool list --feishu-bitable <app-token> --table <table-id>
```

输出：

```text
record-id       status      title
recxxx          待澄清      Add coupon eligibility check
```

从记录创建 demand：

```powershell
devflow intake --feishu-bitable <app-token> --table <table-id> --record <record-id>
```

同步回记录：

```powershell
devflow sync --feishu-bitable <app-token> --table <table-id> --record <record-id> --demand <id>
```

### 9.3 状态映射

默认状态映射：

```text
created              -> 待澄清
context_loaded       -> 已加载上下文
requirements_review  -> 待确认需求
plan_review          -> 待确认方案
implementation       -> 开发中
mr_review            -> 评审中
verification         -> 验收中
closeout             -> 待结项
completed            -> 已完成
failed_quality_gate  -> 质量门失败
blocked_need_platform -> 平台阻塞
```

如果配置中定义 `status_map`，使用配置覆盖默认值。

## 10. 平台鉴权与 Doctor

### 10.1 配置原则

`.devflow/config.yaml`：

- 可以存 app id。
- 可以存默认 repo。
- 可以存飞书字段映射。
- 不能存 token、secret、cookie。

环境变量：

```text
GITHUB_TOKEN
FEISHU_APP_ID
FEISHU_APP_SECRET
```

### 10.2 Doctor 命令

```powershell
devflow doctor --platform github
devflow doctor --platform feishu
devflow doctor --platform all
```

GitHub doctor 检查：

- `GITHUB_TOKEN` 是否存在。
- 能否读取默认 repo。
- 能否读取 issue。
- 能否创建 issue comment，默认用 dry-run 检查权限，不真实写入。

飞书 doctor 检查：

- `FEISHU_APP_ID` 是否存在。
- `FEISHU_APP_SECRET` 是否存在。
- 能否获取 tenant access token。
- 如果传入 doc token，检查能否读取文档。
- 如果传入 bitable app/table，检查能否读取记录。

### 10.3 能力模型

```go
type PlatformCapability string

const (
	CapabilityReadIntake PlatformCapability = "read_intake"
	CapabilityWriteSync  PlatformCapability = "write_sync"
	CapabilityReadPool   PlatformCapability = "read_pool"
	CapabilityWritePool  PlatformCapability = "write_pool"
)
```

每个 adapter 声明所需能力。doctor 输出：

```text
github:
  read_intake: pass
  write_sync: pass

feishu:
  read_intake: pass
  read_pool: pass
  write_pool: missing permission
```

## 11. 事件与审计

每次平台读取写入都要记录事件。

GitHub Issue intake：

```json
{
  "type": "platform.intake_fetched",
  "data": {
    "provider": "github",
    "kind": "github_issue",
    "external_id": "owner/repo#123",
    "url": "https://github.com/owner/repo/issues/123"
  }
}
```

飞书文档 intake：

```json
{
  "type": "platform.intake_fetched",
  "data": {
    "provider": "feishu",
    "kind": "feishu_doc",
    "external_id": "<doc-token>"
  }
}
```

平台同步：

```json
{
  "type": "platform.sync_posted",
  "data": {
    "provider": "github",
    "target": "issue_comment",
    "external_id": "owner/repo#123",
    "stage": "closeout"
  }
}
```

错误事件：

```json
{
  "type": "platform.sync_failed",
  "data": {
    "provider": "feishu",
    "external_id": "<record-id>",
    "reason": "missing write_pool permission"
  }
}
```

## 12. 错误处理

### 12.1 网络错误

平台网络错误不改变 demand 状态，除非用户正在执行必须依赖平台的命令。

例如：

```powershell
devflow sync --github-issue ...
```

如果失败，命令返回非零，写入 `platform.sync_failed` event。

### 12.2 权限错误

权限错误要包含：

- 缺哪个环境变量。
- 缺哪个平台能力。
- 用户应该运行哪个 doctor 命令。

错误示例：

```text
feishu doc intake requires FEISHU_APP_ID and FEISHU_APP_SECRET; run devflow doctor --platform feishu
```

### 12.3 字段映射错误

飞书多维表格字段缺失时，返回：

```text
feishu bitable field mapping missing required field "title" (configured as "需求标题")
```

### 12.4 重复同步

GitHub Issue sync 使用 marker 去重。飞书多维表格 sync 使用 record id 更新，不创建新记录。

## 15. 验收标准

v0.2 完成时，必须满足：

- GitHub Issue 可以创建 Devflow demand。
- GitHub Issue comments 可以进入 `intake.md`。
- Devflow 可以同步 closeout 到 GitHub Issue comment。
- 飞书文档可以创建 Devflow demand。
- 飞书多维表格可以列出需求、创建 demand、同步状态。
- GitHub 和飞书 doctor 能报告关键权限状态。
- 默认 release-readiness 不需要真实 token。
- 所有平台写操作有 dry-run 或明确显式参数。
- 不提交任何 token、secret、cookie。

## 16. 风险

### 16.1 飞书文档格式复杂

飞书文档块结构比 Markdown 复杂。v0.2 只保留语义内容，不追求格式完全还原。

### 16.2 飞书权限配置复杂

飞书应用权限、文档权限、表格权限可能分散。doctor 必须给出明确缺失项，而不是只返回 API 错误。

### 16.3 GitHub Issue 与 PR 都是 Issue

GitHub PR 也会出现在 issue API 中。intake 必须拒绝 PR 类型 issue，避免把 PR 当需求入口。

### 16.4 同步刷屏

GitHub Issue sync 如果不去重，会产生大量重复评论。marker 更新机制是 v0.2 必需能力。

### 16.5 平台写操作风险

所有写操作必须显式调用，不由 `drive` 或 `console --run-next` 自动触发，除非未来版本设计了单独的确认门。

## 17. 与文章产品的对齐

v0.2 对齐《如何搭建一个端到端业务需求专家 Agent》中的两层：

```text
上下文输入层：
  GitHub Issue、飞书文档、飞书多维表格进入 Devflow。

工具执行层：
  GitHub Issue comment、飞书多维表格状态回写。
```

v0.2 暂不完整覆盖：

```text
真实验收证据：
  日志、监控、API 测试 evidence 仍留给 v0.3。

反馈学习层：
  长期 wiki/codemap 写回和 skill 改进候选仍留给 v0.4。

Agent Team：
  前端、测试、PD Agent 协同仍留给更后版本。
```

## 18. Spec 自查

- 范围聚焦在 GitHub + 飞书，不扩散到其他企业平台。
- 读写边界明确：飞书文档只读，飞书多维表格可写状态，GitHub Issue 可写 comment。
- 认证策略明确：secret 只来自环境变量。
- release-readiness 默认离线。
- 所有写操作都有审计事件。
- 不改 `mr_review` workflow state，保持 v0.1 兼容。
